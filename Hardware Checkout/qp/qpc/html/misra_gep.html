Guideline Enforcement Plan (GEP)

MISRA ComplianceMISRA Compliance Matrix

As recommended in [MISRA2-Comp], this section presents the MISRA Guideline Enforcement Plan (GEP). The plan presented here is centered on automatic guideline enforcement of as many rules as possible by means of a static analysis tool, which is stipulated by [MISRA3], Section 3.2 "Analysis Tools".

Note
The completely automatic enforcement of 100% of the MISRA-C rules is not possible and was never intended. Some of the rules are only enforceable by manual review of the code or checking the compiler/linker tools by other means.

The C99 Standard

[MISRA3] requires choosing a specific C programming language standard (either C90 or C99) and then consistently configure the static analysis tool and all compilers to apply this specific C standard. This MISRA Guideline Enforcement Plan chooses C99 as such C language standard.

Attention
The C language standard chosen for this [MISRA3] compliance is C99 (see also Selected C Language Standard).

PC-lint Plus (PCLP) Static Analysis Tool

This Guideline Enforcement Plan uses PC-Lint-Plus (PCLP), which is a static analysis tool for C and C++ with one of the longest track records and best value of the money in the industry. PC-Lint family of tools have been supporting checks for the MISRA-C guidelines since early 2001, and the company behind the tool is committed to provide ongoing and increasing support for these guidelines.

PC-Lint Plus

Contextual Rule Suppression

Attention
This Guideline Enforcement Plan uses the contextual rule suppression as the central principle. It means that the static analysis tool is configured to suppress diagnostics from known deviations only in a specific, well defined context in which the deviation is permitted. Otherwise, a given MISRA guideline remains in full force and gets checked in all other contexts.

Checking MISRA-C:2012 Compliance in PCLP

The primary way of activating checking for MISRA-C:2012 guidelines in PCLP is via the following configuration files:

  • au-misra3.lnt — checking [MISRA3]
  • au-misra3-amd1.lnt — checking [MISRA3-A1]
  • au-misra3-amd2.lnt — checking [MISRA3-A2]

The level of support provided by PCLP for each of the corresponding MISRA-C:2012 guidelines is described in the following sections of the PC-lint Plus User Manual [PCLP].

  • 11.1.2 MISRA C 2012 Guideline Support Matrix
  • 11.2.2 MISRA C 2012 AMD-1 Guideline Support Matrix
  • 11.3.2 MISRA C 2012 AMD-2 Guideline Support Matrix

Additional PCLP Checks

In addition to the aforementioned MISRA checks, PCLP is configured to perform the following checks

  • au-ds.lnt — [Saks-1998] Dan Saks' recommendations for placing const/volatile
  • au-barr.lnt — [BARR-C:2018] style guidelines (customized by Quantum Leaps)

Configuring PCLP

PCLP has several places where it reads its currently valid options:

  • From special Lint option files (usually called *.lnt)
  • From the command line
  • From within the special lint-comments in the source code modules (not recommended)

The QP/C source code and example application code has been “linted” only by means of the first alternative (option files) with possibility of adding options via command line. The third alternative (lint comments) is not used and Quantum Leaps does not recommend this alternative.

Note
The QP/C source code is free of PCLP comments, which are viewed as a contamination of the source code).

The structure of the PCLP option files used for "linting" QP/C follows exactly the published guidelines for configuring PCLP (See Section 3.2 “Configuration” in the PCLP Manual [10]). The design and grouping of the PCLP options also reflects the fact that static code analysis of a software framework, such as QP/C, has really two major aspects:

  • The QP/C framework source code itself

But eually important and helpful to the users of the framework is providing the infrastructure to effectively analyze

  • The Application code based on the QP/C framework.

With this in mind, Listing 1 shows that the PCLP options are divided into two files qpc.lnt and options.lnt, located in directory qpc/ports/lint-plus. These two files are for analyzing QP/C source code and Application code, respectively.

The std.lnt PLCP file

According to the PCLP configuration guidelines, the file std.lnt file contains the top-level options, which apply to QP/C and are recommended for QP/C Applications.

file: std.lnt
[1] -std=c99 // apply the C99 language standard
-zero(99) // don't stop because of warnings
[2] -passes(2) // make two passes (for better error messages)
[3] -restore_at_end // don't let -e<nn> options bleed to other files
-summary() // output a summary of all produced messages
// globally suppress the following warnings:
[4] -e546 // explicitly taking address of function
[5] -e717 // monocarpic do-while used to group statements
1
PLCP is configured to apply the C99 language standard (PCLP option -std=c99).
2
PCLP is configured to make two passes to perform better cross-module analysis.
3
The important option is -restore_at_end, which has the effect of surrounding each source file with options -save and -restore. This precaution prevents options from “bleeding” from one file to another.
4
Global suppression of the warning about explicitly taking address of a function (to create a pointer-to-function, e.g. handler = &Blinky_on). Being explicit about taking an address of a function is the recommended practice in the Quantum Leaps code.
5
monocarpic do-while used to group statements is used in macros that contain multiple statements. Again, this is a recommended practice to avoid any side effects of "dangling" statements.

The qpc.lnt PLCP file

The most important file for "linting" QP/C applications is the qpc.lnt option file. This file handles all deviations from the MISRA-C:2012 rules, which might arise at the application-level code from the use of the QP/C framework. In other words, the qpc.lnt option file allows clean "linting" of the application-level code, as long as the application code does not violate any of the MISRA-C:2012 rules.

At the same time, the qpc.lnt option file has been very carefully designed not to suppress any MISRA-C:2012 rule checking outside the very specific context of the QP/C API. In other words, the qpc.lnt option file still supports all of the MISRA-C:2012 rule checks that PCLP is capable of performing.

The options.lnt PLCP file

The options.lnt option file handles all deviations from the MISRA-C:2012 rules, which might arise at the QP/C source code. These are deviations that are contained within QP/C and generally don't propagate to the QP/C Applications.

The lin.bat batch file

The directory <qpc>/ports/lint-plus (see Listing 1) contains also the lin.bat batch file for "linting" the QP/C source code. The lin.bat batch file invokes PCLP and stores the lint output files. As shown in Listing 1, the lint output is collected into four text files lint_qf.log, lint_qk.log, lint_qv.log, lint_qxk.log and lint_qs.log, for QF, QK, QXK, and QS components of the QP/C framework, respectively.

Note
In order to execute the lin.bat file on your system, you might need to adjust the symbol PCLP at the top of the batch file, to the PCLP installation directory on your computer.

The lin.bat batch file invoked without any command-line options checks the code in the default configuration corresponding to both QP/Spy and QUTest support enabled. But the lin.bat batch can also be invoked with the option -uQ_SPY to disable the the QP/Spy instrumentation and additionally -uQUTest to disable in the unit testing support.

According to the PCLP guidelines, the lin.bat configuration file uses two option files: the std.lnt option file discussed before and the misra_pclp-opt "options.lnt" file covers all deviations from the MISRA-C rules in the QP/C source code. Section MISRA Compliance Matrix cross-references all these deviations, while Section misra_dev describes the reasons for deviations in those, very specific contexts.

QS Software Tracing

As described in Chapter 11 of the book “Practical UML Statecharts in C/C++” [PSiCC2], all the QP/C framework contains software tracing instrumentation (see QP/Spy). This instrumentation code is inactive in the Debug and Release build configurations, and only becomes active in the Spy configuration (macro Q_SPY defined).

In the context of MISRA-C compliance it is important to note that, by the nature of software tracing, the QS instrumentation embedded in the QP/C framework contributes disproportionately to the total number of deviations from the MISRA-C rules, both in the QP/C source code and in the application-level code. However, these deviations occur only in the Spy build configuration, which typically is not the code shipped within a product.

Remarks
By the nature of software tracing, the Q_SPY configuration transgresses many more MISRA-C rules than the standard configuration. However, the Q_SPY configuration is seldom used for production code, so the MISRA-C compliance of the QP/C framework should not be judged by the deviations that happen only in the Q_SPY configuration.
Note
Many of the deviations from the MISRA-C:2012 rules reported in the upcoming MISRA Compliance Matrix do not pertain to the production code.
