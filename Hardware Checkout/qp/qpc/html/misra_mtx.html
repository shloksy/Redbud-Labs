Guideline Enforcement Plan (GEP)Deviations in QP/C Source Code

As recommended in [MISRA3], this section presents the MISRA Compliance Matrix, which lists each MISRA-C:2012 rule and indicates the compliance status and how the rule has been checked.

MISRA Compliance Matrix Conventions

The MISRA Compliance Matrix is presented in the tabular form, whereas the meaning of the table columns is as follows:

Directive/Rule

lists the MISRA-C:2012 directive or rule number followed by the rule classification in parentheses:

  • (M) Mandatory — guidelines for wich violation is never permitted
  • (R) Required — guidelines which can only be violated when supported by a deviation defining a set of clear restrictions, requirements and precautions
  • (A) Advisory — recommendations to be followed as far as reasonably practical. Violations are identified but are not required to be supported by deviation.

PCLP

lists whether a given MISRA guideline is checked by PCLP (see also Configuring PCLP). This column might contain the following:

  • Yes — the guideline is checked by PCLP
  • Part — the guideline is partially checked by PCLP
  • Assist — PCLP provides some assistance in checking the guideline
  • No — the guideline is NOT checked by PCLP

QP/C
lists the compliance status of the QP/C source code. This column might contain the following:
  • M — means that the QP/C source code complies with the MISRA guideline by Manual inspection
  • A — means that the QP/C source code complies with the MISRA guideline by Automatic inspection (via PCLP)
  • PQP<num> — denotes a deviation from the MISRA guideline in the QP/C source code with the QP/C Deviation Permit indicated by <num>.
    Remarks
    You can click on the deviation UID to see the permit.
App
lists the deviations of the Application code imposed by the QP/C framework. This column might contain the following:
  • M — means that the QP/C framework does not impose non-compliance of the Application code with the MISRA guideline, but verification of the compliance requires a Manual inspection
  • A — means that the QP/C framework does not impose non-compliance of the Application code with the MISRA guideline and the compliance can be verified by Automatic inspection (via PCLP)
  • PQA<num> — denotes that the QP/C framework does impose a deviation from the MISRA guideline in the Application code, but the deviation is granted a Application Deviation Permit indicated by <num>.
    Remarks
    You can click on the deviation UID to see the permit.
Headline
contains the MISRA guideline headline, as published in [MISRA3].
Remarks
The use of the MISRA-C:2012 guideline headlines in this document has been licensed to Quantum Leaps, LLC by MIRA Ltd. (Ref: SB/VH/Con/315/MISRA/Sept99, January 6. 2015)

Deviation Permit UIDs

The deviation permits referenced in the compliance matrix are identified using Unique Identifiers (UIDs) with the following structure.

UID for a MISRA Directive Deviation Permit: 

+---------------- 1. work item type ('P' for Deviation Permit)
| ++------------- 2. Permit for the 'QP' source code and 'QA' for QP Application code
| || +----------- 3. letter 'D' for Directive
| || | +--------- 4. one-digit Directive category-number
| || | |+-------- 5. separator ('underscore '_')
| || | ||++------ 6. two-digit Directive number
P QP D x_yy

Examples: PQPD4_9

UID for a MISRA Rule Deviation Permit: 

+---------------- 1. work item type ('P' for Deviation Permit)
| ++------------- 2. Permit for the 'QP' source code and 'QA' for QP Application code
| || ++---------- 3. two-digit Rule category-number (e.g.,'08')
| || ||+--------- 4. separator (underscore '_')
| || |||++------- 5. two-digit Rule number
| || |||||
P QP xx_yy

Examples: PQP08_7
